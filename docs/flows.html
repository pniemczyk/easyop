Flows

Easyop::Flow runs a sequence of operations sharing one ctx. Failure in any step halts the chain and triggers rollback of completed steps in reverse order.

Basic Flow

class ProcessCheckout
  include Easyop::Flow

  flow ValidateCart,
       ApplyCoupon,
       ChargePayment,
       CreateOrder,
       SendConfirmation
end

result = ProcessCheckout.call(user: current_user, cart: current_cart)
result.success?  # => true
result.order     # => #<Order id=42>

How Execution Works

For each step in the flow, EasyOp:

  1. Checks skip? — if skip_if is declared and truthy, the step is skipped entirely (not added to the rollback list)
  2. Checks inline lambda guards (if a ->(ctx){} precedes the step)
  3. Creates an instance of the step class
  4. Calls instance._easyop_run(ctx, raise_on_failure: true) — the step runs with full hook and rescue support
  5. On success: calls ctx.called!(instance) to register for rollback
  6. On failure (Ctx::Failure raised): stops the chain, calls ctx.rollback!
# Simplified execution model:
#
# ProcessOrder.call(attrs)
#   └─ new._easyop_run(ctx, raise_on_failure: false)
#         └─ _run_safe
#               └─ prepare { call }  ← Flow::call
#                     ├─ step1: skip? → instantiate → _easyop_run → called!
#                     ├─ step2: skip? → instantiate → _easyop_run → called!
#                     ├─ step3: fails → raises Ctx::Failure
#                     └─ rescue: ctx.rollback! → step2.rollback, step1.rollback

Rollback

Each step can define a rollback method. On failure, completed steps are rolled back in reverse order. Errors inside individual rollbacks are swallowed so all rollbacks run.

class ChargePayment
  include Easyop::Operation

  def call
    ctx.charge = Stripe::Charge.create(amount: ctx.total, source: ctx.token)
  end

  def rollback
    # Called automatically if a later step in the flow fails
    Stripe::Refund.create(charge: ctx.charge.id) if ctx.charge
  end
end

class CreateOrder
  include Easyop::Operation

  def call
    ctx.order = Order.create!(
      user:  ctx.user,
      total: ctx.total,
      charge_id: ctx.charge.id
    )
  end

  def rollback
    ctx.order.destroy! if ctx.order&.persisted?
  end
end
Rollback rules Only steps that completed successfully (i.e., their call returned without failing) are rolled back. The step that fails is NOT rolled back — only the ones that ran before it.

skip_if — Optional Steps

Declare on the operation class when it should be bypassed inside a flow:

class ApplyCoupon
  include Easyop::Operation

  skip_if { |ctx| !ctx.coupon_code? || ctx.coupon_code.to_s.empty? }

  def call
    coupon = Coupon.find_by(code: ctx.coupon_code)
    ctx.fail!(error: "Invalid coupon") unless coupon&.active?
    ctx.total    -= coupon.discount_amount
    ctx.discount  = coupon.discount_amount
  end
end

class ProcessCheckout
  include Easyop::Flow
  flow ValidateCart, ApplyCoupon, ChargePayment, CreateOrder
end

# ApplyCoupon is automatically skipped when coupon_code is absent or blank:
ProcessCheckout.call(user: user, cart: cart)
ProcessCheckout.call(user: user, cart: cart, coupon_code: "")
# Both skip ApplyCoupon entirely — it's never instantiated, never added to rollback list

# With a coupon:
ProcessCheckout.call(user: user, cart: cart, coupon_code: "SAVE10")
# ApplyCoupon runs normally
skip_if is a Flow concern skip_if is evaluated by the Flow runner before instantiating the step. Calling an operation directly — ApplyCoupon.call(...) — bypasses the skip check entirely. The step will always run when called directly.

Lambda Guards (Inline)

Place a lambda immediately before a step in the flow list to gate it. The lambda receives ctx and should return truthy to proceed, falsey to skip:

class ProcessCheckout
  include Easyop::Flow

  flow ValidateCart,
       ->(ctx) { ctx.coupon_code? && !ctx.coupon_code.to_s.empty? }, ApplyCoupon,
       ->(ctx) { ctx.gift_card? }, ApplyGiftCard,
       ChargePayment,
       CreateOrder
end

Choosing between skip_if and lambda guard:

  • Use skip_if on the operation class when the condition is intrinsic to that operation (e.g., "skip when no coupon")
  • Use a lambda guard in the flow when the condition depends on flow context or orchestration concerns

Nested Flows

A Flow can be a step inside another Flow. The inner flow shares the same ctx and participates in rollback normally.

class AuthAndValidate
  include Easyop::Flow
  flow AuthenticateUser, ValidatePermissions
end

class BillingFlow
  include Easyop::Flow
  flow ChargePayment, CreateInvoice
end

class FullCheckout
  include Easyop::Flow
  flow AuthAndValidate,   # nested flow — runs AuthenticateUser, ValidatePermissions
       ValidateCart,
       ApplyCoupon,
       BillingFlow,       # nested flow — runs ChargePayment, CreateInvoice
       SendConfirmation
end

prepare / FlowBuilder

FlowClass.prepare returns a FlowBuilder that lets you attach callbacks before executing the flow. This is the canonical entry point for controller integration.

Block callbacks

ProcessCheckout.prepare
  .on_success { |ctx| redirect_to order_path(ctx.order) }
  .on_failure { |ctx| flash[:error] = ctx.error; redirect_back }
  .call(user: current_user, cart: current_cart, coupon_code: params[:coupon])

Symbol callbacks with bind_with

# Bind a host object (e.g. a Rails controller) to dispatch to named methods:
def create
  ProcessCheckout.prepare
    .bind_with(self)
    .on(success: :order_created, fail: :checkout_failed)
    .call(user: current_user, cart: current_cart, coupon_code: params[:coupon])
end

private

def order_created(ctx)
  redirect_to order_path(ctx.order), notice: "Order placed!"
end

def checkout_failed(ctx)
  flash[:error] = ctx.error
  render :new
end

Zero-arity methods

If the bound method accepts no arguments, ctx is not passed:

def order_created    # no ctx argument
  redirect_to orders_path
end

Multiple callbacks

ProcessCheckout.prepare
  .on_success { |ctx| Analytics.track("checkout", order_id: ctx.order.id) }
  .on_success { |ctx| redirect_to order_path(ctx.order) }
  .on_failure { |ctx| Rails.logger.error "Checkout failed: #{ctx.error}" }
  .on_failure { |ctx| render json: { error: ctx.error }, status: 422 }
  .call(attrs)

.call vs .call! on Flows

Flows support the same two entry points as single operations. The return type depends on the execution mode:

# .call — Mode 1 & 2: returns ctx, never raises on ctx.fail!
result = ProcessCheckout.call(attrs)
result.success?  # check result

# .call — Mode 3 (durable, subject declared): returns FlowRun, not Ctx
flow_run = OnboardSubscriber.call(user: user, plan: :pro)
flow_run.status  # => "running"

# .call! — raises Easyop::Ctx::Failure on failure (Modes 1 & 2)
begin
  result = ProcessCheckout.call!(attrs)
rescue Easyop::Ctx::Failure => e
  e.ctx.error  # => the error message from whichever step failed
end

Mode 1: Sync Chain

When a flow has no subject declaration and no .async step, it runs every operation synchronously inline and returns a Ctx. This is the default behavior described throughout this page.

class ProcessCheckout
  include Easyop::Flow

  flow ValidateCart,
       ApplyCoupon,
       ChargePayment,
       CreateOrder,
       SendConfirmation
end

result = ProcessCheckout.call(user: current_user, cart: current_cart)
# => Easyop::Ctx
result.success?  # => true
result.order     # => #<Order id=42>

Mode 2: Fire-and-Forget Async

When a flow has no subject declaration but at least one step decorated with .async, the flow enqueues the async step via ActiveJob and moves on immediately — it does not wait for the async step to finish. Steps that come after .async in the flow run before the async step executes.

class RegisterAndNotify
  include Easyop::Flow

  flow CreateUser,              # 1. runs inline, right now
       SendWelcomeEmail.async,  # 2. job pushed to queue — flow does NOT wait
       AssignTrial              # 3. runs inline, before SendWelcomeEmail executes
end

# Execution order inside .call:
#   CreateUser      → runs inline
#   SendWelcomeEmail → job enqueued  ← does not block; email not sent yet
#   AssignTrial     → runs inline    ← runs immediately, before the email lands
#
# Later, in a background worker:
#   SendWelcomeEmail → executes
#
ctx = RegisterAndNotify.call(email: "alice@example.com")
# ctx returned now; SendWelcomeEmail has not run yet
ctx.success?   # => true
ctx.trial_set? # => true  (AssignTrial already ran)
When to use Mode 2 Use fire-and-forget when the async step is a side-effect that does not affect the steps that follow it — sending a confirmation email, triggering a webhook, enqueueing a heavy report. If a later step depends on the async step's result, or you need automatic retry and resume, use Mode 3 instead.
Mode 2 restriction on durable-only options Using .on_exception or .tags on a step inside a Mode-2 flow (no subject) raises PersistentFlowOnlyOptionsError. These options are only valid in Mode 3 flows.
Async plugin required for step-builder DSL Operations that use step-builder class methods — .async, .skip_if, .skip_unless, .on_exception, .tags — must have plugin Easyop::Plugins::Async installed. These methods are added by the Async plugin's ClassMethods module. Operations used as plain bare-class steps (no modifiers) do not need the plugin.

Mode 3: Durable Suspend-and-Resume

Add subject :model_name to make a flow durable. The flow suspends at each .async step, persists the ctx to the database, and resumes from that exact point only after the background job completes. Steps that come after .async do not run until the async step has finished.

This is the key difference from Mode 2: in Mode 3, every step sees the completed results of all previous steps, including async ones.

Setup — opt-in requires

require "easyop/scheduler"       # prerequisite — DB-backed scheduler
require "easyop/persistent_flow" # opt-in; raises DurableSupportNotLoadedError if omitted

AR model setup

bin/rails generate easyop:install --durable
bin/rails db:migrate

Declaring a durable flow

class OnboardSubscriber
  include Easyop::Flow

  subject :user   # <-- durability trigger; ctx is persisted to DB

  flow CreateAccount,           # 1. runs inline
       SendWelcomeEmail.async,  # 2. flow SUSPENDS — job scheduled, .call returns here
       SendNudge.async(wait: 3.days)
                .skip_if { |ctx| ctx[:skip_nudge] },  # 4. flow SUSPENDS again (3 days later)
       RecordComplete            # 5. runs inline after SendNudge finishes
end

# Execution timeline:
#   .call:
#     CreateAccount runs inline
#     flow suspends — SendWelcomeEmail job scheduled
#     FlowRun returned immediately with status: "running"
#
#   (background job fires):
#     SendWelcomeEmail executes
#     flow resumes — SendNudge job scheduled 3 days out
#
#   (3 days later, background job fires):
#     SendNudge executes
#     flow resumes
#     RecordComplete runs inline
#     FlowRun status → "succeeded"
#
flow_run = OnboardSubscriber.call(user: current_user, plan: :pro)
# Returns immediately; flow is suspended waiting for SendWelcomeEmail
flow_run.class   # => Easyop::FlowRun (ActiveRecord model)
flow_run.status  # => "running"
flow_run.subject # => the User AR record

Mode 2 vs Mode 3 at a glance

Mode 2 — fire-and-forget Mode 3 — durable
Step after .async waits? No — runs before the async step finishes Yes — suspends until the job completes
Returns Ctx FlowRun (AR model)
Ctx saved to DB? No Yes — after every step
Requires subject? No Yes
Retry / resume on failure? No Yes — on_exception(:reattempt!)
Use when Async is a side-effect; later steps don't need its result Later steps depend on async result, or you need retry / resume

FlowRun lifecycle

flow_run.cancel!  # => status: "cancelled"; cancels any pending scheduled tasks
flow_run.pause!   # => status: "paused"
flow_run.resume!  # => re-advances from the last completed step

Exception policies (Mode 3 only)

flow CreateAccount,
     ChargeCard.on_exception(:cancel!),                            # fail immediately on any error
     SendWelcomeEmail.on_exception(:reattempt!, max_reattempts: 3) # retry up to 3 times

Backward compatibility

include Easyop::PersistentFlow and .start!(attrs) continue to work as deprecated aliases and will be removed in v0.6. Prefer include Easyop::Flow with subject.

Free Composition

Any flow can embed other flows as steps. The outer flow's mode is determined by the highest-ranked mode present across its step list.

Inner flow typeEffect on outer flow
Mode 1 (sync, no subject)Outer is unchanged
Mode 2 (async, no subject)Async steps of inner run inline as a single step in outer
Mode 3 (subject declared)Inner steps are flattened into outer; outer auto-promotes to Mode 3
 
class InnerDurable
  include Easyop::Flow
  subject :user
  flow StepA, StepB.async(wait: 1.day)
end

class Outer
  include Easyop::Flow
  flow Op1, InnerDurable, Op2   # Outer auto-promotes to Mode 3
end

run = Outer.call(user: user)    # => FlowRun (not Ctx)

New Error Classes (v0.5)

Error classWhen it firesFix
DurableSupportNotLoadedError A flow with subject is called but require "easyop/persistent_flow" was not loaded Add require "easyop/persistent_flow" (and require "easyop/scheduler") to your initializer
AsyncFlowEmbeddingNotSupportedError A Mode-2 (fire-and-forget) flow is embedded directly inside another flow's step list in a way that would break step ordering Refactor: extract the inner async steps or promote the outer to Mode 3 with subject
ConditionalDurableSubflowNotSupportedError A durable (Mode 3) subflow is placed after a lambda guard in the outer flow's step list Move the guard logic into the subflow itself using skip_if, or restructure the step list
